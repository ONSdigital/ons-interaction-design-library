---
title: "Documentation guidelines"
description: "Guidance for contributing to the ONS Interaction Design Library"
---

The Interaction Design Library is design-first documentation. It explains the purpose, behaviour and intended use of interaction design components, patterns and templates.

The primary audience is designers, but the documentation also supports developers, content designers, researchers, product teams and other stakeholders by communicating the design intent and expected user experience.

Documentation should help readers understand:
* what the design is
* why it exists
* when it should be used
* how users interact with it

It should not describe how the design is implemented or duplicate information maintained elsewhere.

## Writing documentation

Every page should explain the design from a user's perspective rather than a technical perspective.

Documentation should be clear, concise and focused on helping teams make consistent design decisions.

Each page should provide enough information for someone to understand the component without needing to inspect the Figma file, while relying on Figma as the source of truth for its visual design.

## Documentation structure

Documentation should follow a consistent structure wherever possible.

Most component documentation should include:
* an overview explaining the component
* its purpose and goals
* guidance on when it should and should not be used
* a description of its structure and anatomy
* any supported states
* behaviour and usage rules
* accessibility considerations

Not every page will require every section, but contributors should follow the established structure wherever it makes sense.

## Questions every page should answer

Well-written documentation answers the questions a designer is most likely to ask when using a component, pattern or template.

When writing documentation, consider whether the page clearly answers the following questions:

- **What is it?**  
  Introduce the component, pattern or template and explain its purpose.

- **Why does it exist?**  
  Describe the user need or design problem it addresses.

- **When should it be used?**  
  Explain the situations where it provides the most value.

- **When should it not be used?**  
  Help contributors understand when another solution would be more appropriate.

- **How does it work?**  
  Describe its behaviour, interaction and any important usage rules.

- **How does it support accessibility?**  
  Highlight any accessibility considerations that influence its design or use.

Not every page will need to answer every question explicitly, but contributors should ensure the documentation provides enough information for readers to confidently understand and apply the design.

## Write about the design

Focus on explaining the decisions behind the design rather than describing its appearance.

Good documentation explains:
* why the component exists
* the problem it solves
* how users interact with it
* how it supports accessibility
* how it should behave
* how it relates to other components

The emphasis should always be on helping designers make informed design decisions.

## Avoid implementation details

The Interaction Design Library is not development documentation.

Do not include implementation-specific information such as:
* HTML
* CSS
* JavaScript
* framework-specific guidance
* code snippets
* implementation techniques
* markup terminology

Describe interface elements using design language rather than implementation language so that the guidance remains applicable regardless of the technology used to build it.

## Avoid duplicating information

The Interaction Design Library should remain the authoritative source for design guidance without duplicating information maintained elsewhere.

### Visual specifications

Visual specifications are maintained within the Figma component library and Dev Mode.

Avoid documenting:
* colours
* typography
* spacing
* dimensions
* design tokens
* variables
* measurements
* visual styling values

Where contributors need this information, they should refer to the Figma component library.

### Content guidance

Content guidance is maintained by the Content Design team.

Avoid reproducing content standards, writing guidance or editorial rules within the Interaction Design Library.

Where appropriate, link to the relevant Content Design guidance instead.

## Write for longevity

Documentation should describe information that is unlikely to change.

Design intent, behaviour and usage guidance remain valuable even as visual styles evolve.

By avoiding duplicated specifications and implementation details, the documentation remains accurate and requires less ongoing maintenance.

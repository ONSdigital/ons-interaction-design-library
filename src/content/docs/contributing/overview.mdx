The Interaction Design Library is a shared resource that supports the creation and maintenance of consistent, high-quality interaction design patterns and documentation.

To ensure the library remains accurate, reliable and easy to maintain, all contributions should follow the established processes and standards. This guidance explains how to contribute to the library, from setting up your working environment to proposing, reviewing and publishing changes.

Whether you're correcting a typo, improving documentation or updating a component, every contribution follows the same workflow to ensure changes are reviewed, traceable and aligned with the rest of the library.

## Before you start

Before contributing to the Interaction Design Library, you should:
* Have access to the required tools and services, including GitHub, Figma and Jira.
* Complete the environment setup guide so you can run the documentation locally.
* Be familiar with the library's documentation standards and contribution workflow.
* Discuss significant changes with the appropriate people before beginning work, where necessary.

Making changes through the agreed process helps maintain quality, prevents conflicts and ensures that all contributors have visibility of ongoing work.

## Roles and responsibilities

The Interaction Design Library is maintained collaboratively. Everyone is encouraged to contribute improvements, while following the agreed review and approval process.

Contributors are responsible for:
* Following the documented contribution workflow.
* Ensuring changes are accurate, complete and consistent.
* Updating related documentation where appropriate.
* Responding to feedback raised during reviews.

Reviewers are responsible for:
* Checking that changes meet the library's standards.
* Providing constructive feedback where improvements are needed.
* Approving changes that are ready to be merged.

The library maintainers are responsible for:
* Maintaining the overall quality and consistency of the library.
* Managing releases and major changes.
* Evolving documentation standards and contribution processes over time.

## Contribution principles

Every contribution should follow these principles.

### Keep changes focused

Each contribution should address a single piece of work wherever possible. Smaller, focused changes are easier to review, test and understand.

### Follow the documented process

Always use branches, pull requests and the appropriate review process. Avoid making changes directly to the main branch or the main Figma file.

### Maintain consistency

Follow the existing documentation structure, writing style and design standards. New content should feel like a natural extension of the existing library.

### Document design decisions, not implementation

Documentation should explain the purpose, behaviour and intended use of components and patterns. It should not include implementation details such as HTML, CSS, JavaScript or design token values.

### Keep documentation and design aligned

Where changes affect both documentation and the Figma library, ensure they remain consistent so contributors always have a single, reliable source of truth.

### Leave the library better than you found it

If you notice a small issue while making another change—such as a typo, broken link or outdated reference—consider fixing it as part of your contribution where appropriate.

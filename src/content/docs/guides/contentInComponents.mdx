description: "Explaining why the ONS design system components use example content rather than mirroring live website content."
---

The ONS design library focuses on design patterns, structure, and behaviour, rather than acting as a source of truth for live website content. Components are populated with representative example content to help designers understand how they are intended to be used, not to reflect the exact content users see on the site at any given time.

This distinction is important for the long-term health and usability of the design system.

## Why components do not mirror live site content

Content across the ONS website changes frequently. Navigation labels, footer links, and content groupings evolve in response to user needs, policy changes, and organisational priorities. If the design system attempted to stay aligned with live content, it would quickly become difficult to maintain and easy to misinterpret.

Keeping components in sync with real content would also shift the purpose of the library. Instead of being a stable reference for design decisions, it would risk becoming outdated, misleading, or treated as an authoritative content source when it is not intended to be one.

For these reasons, the design system deliberately avoids reflecting live content verbatim.

## The role of example content

Example content exists to support understanding, not accuracy. It helps designers see how components behave with realistic text, links, and labels, without implying that those values are final or agreed.

In practice, example content is used to:
* demonstrate layout, hierarchy, and emphasis
* show typical content length and density
* illustrate how a component behaves with real-world information

While this content is written to feel plausible and meaningful, it should always be treated as illustrative.

## Design system versus content ownership

The design system and the live website serve different roles.

The design system defines how things work: how components are structured, how they scale, how they respond, and how they fit together consistently. Content design and product teams define what appears within those structures at any moment in time.

Navigation structures, link labels, and content groupings are therefore owned and maintained outside the design system. The system provides the framework that allows those decisions to be implemented consistently, without needing to update the components themselves every time content changes.

## Using realistic content responsibly

Where possible, components use realistic language rather than abstract placeholders. This makes them easier to understand and more useful in day-to-day design work. However, this does not imply that the content shown is complete, final, or approved.

When using components, designers should replace example content with real content as early as possible, and refer to live sources or content designers when accuracy matters. The design system should be seen as a guide to structure and behaviour, not a reference for agreed wording.

## Summary

Components in the design library are intentionally decoupled from live content. This allows the system to remain stable, maintainable, and focused on design quality, while content continues to evolve independently.

This approach helps ensure the design system remains a reliable tool for designers without becoming brittle, outdated, or misleading over time.
